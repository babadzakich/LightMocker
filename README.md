# LightMocker

LightMocker — это легковесный Java-фреймворк для создания мок-объектов, разработанный в рамках курса по проектированию программного обеспечения (НГУ). Фреймворк предоставляет функционал для стаббинга, верификации и частичного мокирования (spies).

## Отличия от стандартной постановки задачи

LightMocker расширяет базовые требования за счет следующих возможностей:
1.  **Поддержка Spies (Частичное мокирование)**: Возможность оборачивать реальные объекты. Если поведение метода не переопределено (stubbed), вызывается оригинальный метод реального объекта.
2.  **Мокирование конкретных классов**: Поддержка не только интерфейсов, но и обычных классов (включая те, у которых нет конструктора по умолчанию) через инструментацию байт-кода.
3.  **Smart Method Resolution**: Автоматическое определение нужного метода по имени и переданным аргументам в `withArgs(...)`, что избавляет пользователя от необходимости явно указывать типы параметров (`.class`) в большинстве случаев.
4.  **Типизированный Fluent API**: Использование Generic-типов в построителях стабов для обеспечения проверки типов на этапе компиляции.

## Дизайн API

Дизайн ориентирован на читаемость (Fluent Interface) и безопасность типов.

### 1. Точки входа (Entry Points)
Основное взаимодействие происходит через класс `ru.nsu.CustomUltimateMocker`.
- `create(Class<T>)`: Создание чистого мока.
- `spy(T object)`: Создание прокси над реальным объектом.
- `setup(...)`: Начало конфигурации поведения (стаббинг).
- `verify(...)`: Начало проверки вызовов.

### 2. Использование Generics
Класс `SetupBuilder<R>` параметризован типом возвращаемого значения метода. Это позволяет методам `.returns(R value)` принимать только те объекты, которые совместимы с сигнатурой метода, минимизируя ошибки в тестах.

### 3. Механизм "Smart Setup"
Пользователь может вызвать `setup(mock, "methodName")` без указания типов аргументов. Фреймворк соберет все подходящие методы (кандидаты) и окончательно выберет нужный только после вызова `.withArgs(...)`, сопоставив типы фактически переданных объектов с сигнатурами методов.

## Архитектура и реализация

### 1. Инструментация и проксирование
Фреймворк использует гибридный подход для создания прокси-объектов:
- **JDK Dynamic Proxy**: Используется для интерфейсов как наиболее легковесное стандартное решение.
- **ByteBuddy**: Используется для мокирования классов. Фреймворк генерирует подкласс целевого типа "на лету", перехватывает все вызовы и делегирует их в `InvocationHandler`. Это позволяет мокировать даже те классы, которые требуют сложной инициализации.

### 2. Управление состоянием и Identity
Критически важной деталью реализации является использование **`IdentityHashMap`** в реестрах вызовов и стабов.
- **Почему это важно?** Обычный `HashMap` при вставке или поиске вызывает методы `hashCode()` и `equals()` у ключа (мок-объекта). Если бы мы использовали обычный `HashMap`, это привело бы к регистрации "лишних" вызовов этих методов или даже к бесконечной рекурсии, если `hashCode()` застаблен. `IdentityHashMap` сравнивает объекты строго по ссылке (`==`), игнорируя переопределенные методы объекта.

### 3. Механизм Spies
В `MockProxy` хранится ссылка на `target` (реальный объект). Если при вызове метода в `StubRegistry` не найдено подходящего правила, прокси проверяет наличие `target`:
- Если `target` есть (это Spy) — вызывается оригинальный метод через Reflection.
- Если `target` нет (это Mock) — возвращается значение по умолчанию для типа.

### 4. Передача контекста через ThreadLocal
Для реализации краткого синтаксиса `when(mock.method()).returns(...)` используется `ThreadLocal` хранилище последнего вызова. Это позволяет связать вызов метода у мока с последующим вызовом в DSL, не передавая состояние явно.

## Примеры использования

### Базовый стаббинг (с типизацией)
```java
List<String> mock = create(List.class);
// Типизация через возвращаемое значение метода
setup(mock, "get").withArgs(0).returns("Hello"); 
```

### Использование Spy
```java
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);

spyList.add("A"); // Реальный вызов
setup(spyList, "size").returns(100); // Переопределенный вызов

System.out.println(spyList.get(0)); // "A"
System.out.println(spyList.size());   // 100
```

### Проверка порядка (In-Order)
```java
InOrder inOrder = inOrder(mockA, mockB);
inOrder.verify(mockA, "start").once();
inOrder.verify(mockB, "process").once();
```
